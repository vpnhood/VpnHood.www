# SEO & Semantic-HTML Conventions

Shared conventions for keeping our static Jekyll marketing sites accessible and
SEO-correct. **This file is the single source for the SEO + semantic-HTML
standard, and it is intentionally identical in both repos — `VpnHood.www` and
`paymenthood-www`.** Each site's `CLAUDE.md` only links here. The rules are
**binding for any markup change** — pages, shared includes (`header.html`,
`footer.html`, `faq.html`, and any product/breadcrumb templates), and components.

Edit this file in **both repos together** so they never drift. Where the two
sites genuinely differ (only the title mechanism, §3), the difference is called
out in a per-site note — everything else applies to both verbatim.

Core principle, applied throughout: **if a piece of text is not part of the
page's content outline, it is not a heading** — and **visual size is a CSS
concern, never a reason to pick a heading level.**

---

## 1. SEO tags, JSON-LD & site metadata

How a site emits its meta / OpenGraph / structured data. It's configured
centrally — don't hand-add per-page tags that seo-tag already emits.

- **`{% seo %}`** (jekyll-seo-tag) in `header.html` emits the meta description,
  canonical, OpenGraph, Twitter Card, and a WebSite/WebPage JSON-LD. See §3 for
  the per-site title handling. **Don't** add a manual
  `<meta name="description">` / `<link rel="canonical">` — that duplicates
  seo-tag.
- Site-wide values live in `_config.yml`: `url`, `title`, `description`, `logo`,
  `twitter`, `social.links`, and a `defaults` `image:` (the default OG/Twitter
  card; override per page with front-matter `image:`).
- A hand-written **Organization** JSON-LD lives in `header.html` (seo-tag already
  emits WebSite — don't duplicate it). The Organization `logo` should be the
  **brand wordmark logo**, per Google's logo guidelines — not a mascot/app icon.
- `robots.txt` points at the site's `/sitemap.xml`, generated by
  `jekyll-sitemap` (never hand-write a sitemap — a manual file silently disables
  the plugin). Utility pages (`404.html`, `thank-you.html`) carry
  `noindex: true` + `sitemap: false`; `header.html` renders the `noindex` robots
  meta from `page.noindex`.
- Performance: `<link rel="preconnect">` to font hosts; per-page LCP image
  preload via front-matter / inlined critical CSS for the hero.

### Generated structured data — never hand-write per page

Beyond Organization/WebSite, any of these schema types must come from a **shared
include**, never copy-pasted per page (hand-written JSON-LD drifts out of sync
and produces duplicate/conflicting entities):

- **FAQPage** — `_includes/faq.html` (renders the accordion **and** the schema
  from a `{q, a}` list).
- **SoftwareApplication** — e.g. `_includes/software-application.html` (one
  canonical listing per product; product pages only, never on sub/install pages).
- **BreadcrumbList** — e.g. `_includes/breadcrumb.html` (derived from the page
  URL).

If a site doesn't have a given page type yet, the rule still stands: when you add
one, generate its schema from an include.

### Favicons & app icons

Drive the whole icon set from **one high-resolution square source image**; never
ship placeholder or 1×1 icons (a real, historic bug here: a manifest whose only
icon was a 1×1 transparent base64 PNG).

- From the single source, produce: a `favicon` (16/32 px, or a multi-size
  `.ico`), an `apple-touch-icon.png` (180×180), and PWA manifest icons at
  **192×192** and **512×512** (mark the 512 `"purpose": "any maskable"`).
- `header.html` links them: `<link rel="icon" … sizes>`,
  `<link rel="apple-touch-icon" …>`, and `<link rel="manifest" …>`. The
  `site.webmanifest` `icons:` array references the real 192/512 PNGs.
- The square app-icon source is for favicons/app icons only — it is **not** the
  OG card (a purpose-built 1200×630 image) and **not** the Organization `logo`
  (the wordmark).

## 2. Heading outline — one H1, no skipped levels

Every page must form a clean outline: `h1 → h2 → h3 …`, descending one level at a
time. Going back *up* any number of levels is fine; skipping *down* a level
(e.g. `h1` then `h3`) is not.

- **Exactly one `<h1>` per page** = the page's main title. On secondary pages
  rendered through a shared header that prints `page.title` as the visible
  heading, that header **is** the H1 — so in-content section titles start at
  `<h2>`. Never a second H1.
- **Don't skip levels.** A section title under the page H1 is `<h2>`; a card or
  item under that section is `<h3>`; and so on.
- **Tag = semantic level; size = class.** Decouple appearance from level using
  Bootstrap's `.h1`–`.h6` size classes. `<h3 class="h5">Foo</h3>` is level 3 in
  the outline but *looks* like an h5. This is the mechanism for re-leveling a
  heading **without changing how it looks** — always carry the original size as
  a `.hN` class when you change a tag (unless a class already sets the size).
- **Non-outline text must not be a heading.** Decorative, boilerplate, brand, or
  navigation text is not content outline → use `<div>`/`<p>`/`<span>`. The
  canonical example, fixed in both repos: the **footer column labels** (Quick
  Access / Solutions / Legal / Support …) are navigation repeated on every page;
  as `<h6>`/`<h2>` they dilute each page's real content headings, so they are
  `<div>` (keep the styling class, e.g. `wgs-title`/`footer-module-title`).

### Section markup pattern

The "section head" pattern must map to the outline like this:

| Visual element            | Element / class                          | Outline role        |
|---------------------------|------------------------------------------|---------------------|
| Eyebrow / kicker          | `<span>`/`<p>` (e.g. in `.section-label`/`.section-head`) | **not a heading** |
| Section title             | `<h2 class="section-title …">`           | section (h2)        |
| Card / feature / sub-item | `<h3 …>` (+ `.hN` size class as needed)  | subsection (h3)     |

The eyebrow is a label for the title, not a separate outline node, so it is a
`<span>`/`<p>`, never a heading.

### FAQ component (`_includes/faq.html`)

The question heading level is context-dependent and driven by an assign at the
top of the include — **don't hardcode the question tag**:
- default (FAQ section sits under a page H1 with its own `<h2>` section title):
  each question renders `<h3>`.
- when the FAQ supplies the page's own top heading (`heading=false`): each
  question renders `<h2>`.

### Verify before committing markup changes

After `bundle exec jekyll build`, every page in `_site` must be single-H1,
first-heading-H1, and skip-free. Quick audit:

```bash
python - <<'PY'
import re, glob, os
bad = 0
for f in sorted(glob.glob("_site/**/*.html", recursive=True)):
    if os.sep+"assets"+os.sep in f: continue
    hs = [int(m.group(1)) for m in re.finditer(r'<h([1-6])\b', open(f, encoding="utf-8").read())]
    if not hs: continue
    v = []
    if hs[0] != 1: v.append("first=h%d" % hs[0])
    if hs.count(1) != 1: v.append("%d h1" % hs.count(1))
    for i in range(1, len(hs)):
        if hs[i] > hs[i-1] + 1: v.append("skip h%d->h%d" % (hs[i-1], hs[i]))
    if v: bad += 1; print(f, v)
print("pages with violations:", bad)   # must be 0
PY
```

## 3. Title tags — brand on every page

**The principle (both sites): the brand name must appear in every page's
`<title>` (and therefore in the `og:title`/Twitter title derived from it).** A
page's `title:` front matter is the single source for all three. The brand
belongs in the title tag, **not** in a duplicate H1 — one descriptive H1 + brand
in the title + the Organization JSON-LD is the SEO-optimal arrangement; don't
reintroduce a brand H1 to "boost" the keyword.

**The two sites wire the brand into the title differently — match the site you're
editing:**

| Site             | seo-tag call            | `<title>` source                | Brand in `title:` front matter?            |
|------------------|-------------------------|---------------------------------|--------------------------------------------|
| **VpnHood.www**  | `{% seo title=false %}` | a hand-written `<title>{{ page.title }}</title>` above the tag | **Yes** — every `title:` must contain `VpnHood!` (seo-tag won't append it). Convention: `Descriptive Keywords \| VpnHood!`; home is brand-first `VpnHood! — Free, Secure & Open-Source VPN`. |
| **paymenthood-www** | `{% seo %}` (default) | seo-tag emits it and **appends** ` \| PaymentHood` (the `site.title`) | **No** — `title:` must NOT contain the brand, or it would render doubled (`… \| PaymentHood \| PaymentHood`). |

Why the difference: VpnHood uses `title=false` deliberately for a brand-first
home title and full separator control; PaymentHood lets seo-tag append the brand
so it can never be forgotten. Both yield the brand in every `<title>` — don't
"unify" one onto the other without a deliberate, tested migration.

Audit titles for the brand (run in the relevant repo; set `BRAND`):

```bash
BRAND="VpnHood"   # or "PaymentHood"
for f in $(find . -name '*.html' -not -path './_site/*' -not -path './_archive/*' -not -path './assets/*'); do
  t=$(awk '/^---/{n++;next} n==1 && /^title:/{sub(/^title:[ ]*/,"");print;exit}' "$f")
  # VpnHood: every title must contain the brand. PaymentHood: seo-tag appends it,
  # so check the *rendered* _site/<page> <title> instead of front matter.
  echo "$f -> $t"
done
```

## 4. Image & inline-SVG accessibility

Every image and inline icon is either **decorative** — it adds nothing a screen
reader needs because adjacent text already conveys it — or **meaningful**,
carrying information not available from nearby text. Mark each accordingly.

### Inline `<svg>`

Every inline `<svg>` is either decorative or meaningful — mark it accordingly:

- **Decorative icon** (redundant with adjacent visible text): add
  `aria-hidden="true"` to the `<svg>`. This is the default for the icons on these
  sites (feature bullets, chevrons, check marks, contact-card glyphs, etc.).
- **Icon-only link/button** (an `<a>`/`<button>` whose only content is an SVG,
  with no visible text): put an `aria-label` on the **link/button** (e.g.
  `aria-label="…on LinkedIn"`); the inner `<svg>` stays `aria-hidden`. Without
  this, a screen reader announces only the URL.

A bare `<svg>` with neither attribute is a defect.

### `<img>` alt text

Every `<img>` needs an `alt` attribute; its **value** depends on the role:

- **Informative image** (a logo in a label-less slider, a screenshot, a diagram
  that conveys content): give a short, specific `alt` describing what it shows or
  names — never the file name.
- **Decorative image** (a brand/product logo or icon sitting **right next to a
  heading or label that already names it**, hero artwork, background shapes,
  mascots): use an **empty `alt=""`** so assistive tech skips it. A non-empty
  `alt` here is announced **on top of** the adjacent text — the duplication the
  [axe `image-redundant-alt`](https://dequeuniversity.com/rules/axe/4.11/image-redundant-alt)
  rule flags. The canonical case on these sites, fixed in both repos: a
  product/gateway logo immediately followed by an `<h1>`/`<h2>`/`<h3>` of the same
  name → the logo is `alt=""`.
- **Never drop `alt` entirely.** A missing attribute makes some screen readers
  read the file path aloud. Decorative means `alt=""`, not *no* `alt`.

Rule of thumb: if removing the image would lose information that isn't in the
surrounding text, it's informative — describe it; otherwise it's decorative —
`alt=""`.

## 5. Lists for groups of links

A group of related links is a list — `<ul><li>` — not loose `<a>`s in a wrapper.
Put layout utilities on the `<ul>` and strip the default list chrome:

```html
<ul class="d-flex align-items-center gap-3 list-unstyled mb-0">
  <li><a aria-label="…on LinkedIn (opens in new tab)" …>…</a></li>
  …
</ul>
```

Applies to footer nav columns, social-icon rows, and menu link groups. Screen
readers then announce "list, N items," which is the correct grouping.

## 6. External links that open a new tab

Any link with `target="_blank"` must announce the new tab/window (WCAG 2.4.4 /
technique G201) — otherwise the context change is silent for assistive tech.

- **Icon-only links** (social buttons, etc.) → append ` (opens in new tab)` to
  the existing `aria-label` (e.g. `aria-label="…on LinkedIn (opens in new tab)"`).
- **Text links** → add a `<span class="visually-hidden"> (opens in new tab)</span>`
  (Bootstrap utility, available from the vendor bundle) right after the visible
  label.

Screen-reader-only — no visual change. Pair `target="_blank"` with
`rel="noopener"` for security while you're there. Keep this in sync whenever you
add a `target="_blank"` link (pages, `header.html`, `footer.html`, templates).

## Why this matters (rationale, in one place)

- Centralizing meta / OG / JSON-LD in seo-tag + `_config.yml` + shared schema
  includes keeps every page's metadata consistent and avoids duplicate or
  conflicting tags that confuse crawlers.
- A single descriptive H1 + clean outline is how search engines and assistive
  tech read a page's structure; duplicate/boilerplate/skipped headings muddy it.
- Repeating generic navigation labels (footer columns) as headings on every page
  makes them compete with real content topics — boilerplate dilution. Demote to
  non-headings.
- The brand keyword earns the most in the `<title>`; a hidden duplicate H1 earns
  almost nothing and breaks the outline.
- A real icon set (favicon + apple-touch + maskable 192/512) makes the site
  installable and recognizable in tabs, bookmarks, and home screens; a 1×1
  placeholder is an invisible defect.
- `aria-hidden`/`aria-label` on SVGs and list semantics for link groups are
  low-cost, high-value a11y wins that also keep the DOM clean for crawlers.
- Decorative `<img alt="">` (and a specific `alt` on informative images) stops a
  screen reader from reading a logo's name on top of the heading beside it, or
  announcing a raw file path — the same decorative-vs-meaningful call as SVGs.
- Announcing `target="_blank"` lets screen-reader users anticipate the context
  switch instead of being silently moved to a new tab.
